The current design of the system uses a polling mechanism to manage communication chunks between the different components. This was done on purpose to avoid over-reliance on persistent connections when hosts and clients may experience network disruptions (eg mobile devices on the go). That said, it's also extremely inefficient and laggy when network connectivity is not an issue.

The alternate approach would be something like Web Sockets where hosts and clients maintain open connections to the server and input/output chunks are streamed. I'd like to explore how we could add support for this approach without sacrificing the resiliency designed into the polling mechanism. Please read through the current architecture carefully and make a proposal for extending the communication to manage persistent connections.

---

Feedback as I read through the plan:

* This project is still in prerelease, so API breaking changes are completely fine (there will be no client migration problems)
* I don't think we need to maintain independent polling / websocket / hybrid modes and can simply consolidate all connectivity to hybrid
* In the `Connection Lifecycle`, I see `WS Upgrade Request/Response`, but I'm not clear which messages that corresponds to in the above message types
* When attempting to upgrade from polling to WS, 5 minutes is too long to remain degraded. We should be able to do the upgrade check much more frequently if we can keep it lightweight.
* I'm not intimately familiar with how WebSockets handles packet loss, but since the connection is persistent and the frame payloads expect valid json, I would imagine we could hit a situation where the stream gets into a bad state (partial frame followed by good frames, or just garbage stream). We need to figure out how to handle this w.r.t. the client connection, likely by detecting this condition and downgrading to polling or resetting the WS connection.
    * Related: currently, the stream purging logic relies on the server's view of which chunks have been delivered to the clients (handlers.go:204), but there is no receipt mechanism for clients to ack receipt of those frames. We don't need to solve this now, but we should keep a TODO to handle this or explicitly document the best-effort design of the architecture where it's expected that clients may miss some chunks.
* In the `Security Considerations` section, it notes `Client ID extracted from cert CN`. We attempted to refactor this earlier so that `client_id` was always explicitly part of the API rather than inferred from TLS. It's possible that refactor didn't complete correctly, but it _should_ be the case that `client_id` is an application-level data element and mot a transport-level data element, so it should not rely on the transport-layer security.
* There's design around rate limiting in the `Security Considerations` section. The nature of this tool _requires_ trusted clients (thus mTLS) due to the level of privilege the client gets on the host machine, therefore we don't need to mitigate for malicious rate abuse, but rather inadvertent abuse due to incorrect usage (eg wrapped commands that refresh too frequently). As such, I think we should _not_ implement rate limiting now, but should add it as a TODO in the event that it becomes a problem even with trusted clients.
* I like the introduction of `load` testing, but I want to be careful about mixing this with other styles of testing. We should consider adding different `make` targets for the different flavors of testing.
* In the logging section, we should ever log to WARNING in a case that the user might need to intervene. Many wrapped commands get badly broken by these warning messages due to their TUI states, so the only currently listed events that should be logged on WARNING would be `Invalid messages` (also possibly not if handled gracefully) and `Authorization failures` since these are unrecoverable.
